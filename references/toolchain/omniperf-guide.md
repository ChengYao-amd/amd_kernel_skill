# ROCm Compute Profiler (rocprofiler-compute) Guide

The **on-chip metric collection + report analysis** tool **Omniperf** for MI series GPUs has been officially renamed to **ROCm Compute Profiler**; the executable is **`rocprof-compute`** (repository [ROCm/rocprofiler-compute](https://github.com/ROCm/rocprofiler-compute)). **The workflow remains the same as the legacy version**: first **profile** to collect data, then **analyze** to view metrics; changes are mainly in **the command name, analysis entry points (CLI / Web), and underlying rocprof version options**.

Official documentation: [ROCm Compute Profiler](https://rocm.docs.amd.com/projects/rocprofiler-compute/en/latest/index.html).

## Core Features (aligned with official description)

- **One-shot automated collection**: Polls required **PMC** through preset **perfmon input files**, covering counters needed for each section of the analysis report, eliminating the need to manually write full counter lists. See [Profile mode -- Profiling routine](https://rocm.docs.amd.com/projects/rocprofiler-compute/en/latest/how-to/profile/mode.html).
- **Filtering for speedup**: Can narrow the collection scope by **kernel substring** (`-k`), **dispatch number** (`-d`), **hardware report block** (`-b`, corresponding to report sections/metric blocks).
- **Standalone Roofline**: Use `--roof-only` to run only roofline-related tests; default profile often includes a **roofline benchmarking phase** (unless `--no-roof`). Empirical roofline PDF and other outputs are documented in the official *Profile mode*.
- **Analysis methods**: **Terminal CLI**, **Grafana dashboard**, **Standalone GUI**, **TUI**, and other approaches ([Analyze mode](https://rocm.docs.amd.com/projects/rocprofiler-compute/en/latest/how-to/analyze/mode.html)).

## Basic Workflow (new command name)

```bash
# 1. Collection (example: name the workload, run the command)
rocprof-compute profile --name workload_name -- python run_kernel.py

# 2. Command-line analysis (path points to the workload directory generated by profile)
rocprof-compute analyze -p workload_name/MI300X --cli

# 3. Graphical analysis (choose one; subcommands per `rocprof-compute analyze -h` and docs)
#    - Grafana dashboard: see official Analyze mode -> Grafana GUI analysis
#    - Standalone GUI / TUI: see same section
```

**Notes**:

- The target subdirectory name varies by **SoC**, commonly **`MI200` / `MI300X` / `MI300A`** etc. (see official [compatible accelerators](https://rocm.docs.amd.com/projects/rocprofiler-compute/en/latest/reference/compatible-accelerators.html)).
- The underlying **ROCProfiler** actually invoked may be **rocprofv1 / rocprofv3** etc.; the installation package or `--help` will indicate the **Profiler choice**.

Common auxiliary commands:

```bash
rocprof-compute profile -h
rocprof-compute profile --list-metrics    # List report block numbers for -b filtering
rocprof-compute analyze -h
```

## Speed-of-Light (SoL) Analysis Panel

In the **`--list-metrics`** output, **System Speed-of-Light** corresponds to the **"speed-of-light" class metrics** in the report: comparing the current kernel's key throughput (such as **VALU FLOPs, MFMA FLOPs, memory hierarchy bandwidth**, etc.) against **empirical roofline** or **peak values**, for quickly determining **whether compute is fully utilized, whether it is memory-bound, and whether Matrix Cores are being used**.

Typical subsections include (numbering varies by version; refer to local `--list-metrics` output):

- **2.1 Speed-of-Light**: Overview of **GPU / engine** utilization and bottleneck direction.
- **2.1.0 VALU FLOPs**, **2.1.1 VALU IOPs**: Throughput related to non-MFMA vector instructions.
- **2.1.2 MFMA FLOPs (F8)** etc.: Low-precision Matrix Core paths.
- Plus **HBM/L2/LDS** related bandwidth and utilization entries (continuing in the numbered list).

**Reading recommendation**: Start with the **SoL** overview, then drill down into **MFMA / VALU / memory hierarchy**; cross-verify with the **Roofline PDF** and `pmc_perf.csv`.

## Key Metrics (healthy ranges are rough guidelines only)

| Panel/Category | Metric | Healthy (indicative) | Needs attention (indicative) |
|----------------|--------|----------------------|------------------------------|
| Speed-of-Light | GPU / engine effective utilization | Depends on workload | Persistently very low may indicate insufficient launches or MFMA not being used |
| Speed-of-Light | Memory bandwidth utilization | Should be high when memory-bound | Claims memory-bound but bandwidth very low -> coalescing/concurrency issue |
| Compute | MFMA utilization | Should be high for GEMM-class workloads | Too low -> tile/precision/instructions not going through MFMA |
| Compute | VALU utilization | Depends on algorithm | Abnormally high but MFMA low -> possibly not using matrix cores |
| Memory | L2 hit rate | Higher is better (depends on access pattern) | Too low -> poor reuse or working set too large |
| Memory | LDS bank conflict | Low | High -> LDS pattern or bank contention |
| Occupancy | Actual occupancy | Depends on registers/LDS | Too low -> resource limitation or block configuration |
| Occupancy | Scratch / spill metrics | 0 is ideal | Non-zero -> register pressure too high or spilling |

Exact column names depend on the **analyze** output and **GUI**.

## Bottleneck Decision Tree (simplified)

```
GPU/SoL shows very low overall utilization?
+-- Yes -> launch density, grid too small, excessive synchronization, CPU side not feeding GPU
+-- No -> Check memory bandwidth vs compute
    +-- Memory bandwidth high and close to roofline -> memory-bound: coalescing, vectorization, tiling, L2-friendly
    +-- Memory bandwidth not high -> Check MFMA/VALU
        +-- MFMA high -> compute/MFMA path dominant, try mixed precision, algorithmic changes
        +-- MFMA low but VALU/mem-access abnormal -> instruction mix, dependencies, not using MFMA, check compilation and algorithm mapping
```
