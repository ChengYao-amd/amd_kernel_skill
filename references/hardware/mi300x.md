# MI300X (GFX942 / CDNA3) Optimization Guide

Same-architecture variant: **MI325X** is also GFX942 / CDNA3, sharing the same CU count and ISA as MI300X; the main differences are HBM capacity and media (see "Variant Notes" below).

## Partition and NPS Modes (Multi-XCD / NUMA)

MI300X-class **8 XCD** devices support multiple **partition** and **NPS (NUMA Per Socket)** combinations, affecting visible GPU topology, isolation, and memory access locality:

| Mode | Description | Typical Use Case |
|------|-------------|------------------|
| **SPX** (Single Partition eXperience) | **8 XCDs** exposed as a **single GPU** | **Default**; maximizes single-card compute and unified address space |
| **DPX** (Dual Partition) | **2x virtual GPUs**, each with **4 XCDs** | **Inference serving** and other scenarios requiring dual-tenant or dual-instance isolation |
| **CPX** (Compute Partition eXperience) | **8x independent partitions** | **Strong isolation** (hard partitioning for multiple jobs / multi-tenant) |

**CDNA4 Note**: **GFX950** adds **QPX** (**4** partitions x **2** XCD) and other modes, and the **NPS** options with **IOD=2** layout differ from CDNA3; see [`mi355x.md`](mi355x.md) for details.

**NPS (NPS1 / NPS2 / NPS4)**: Controls **memory interleaving** strategy and intra-socket NUMA behavior; combinations with partition modes alter bandwidth and latency characteristics.

- **CPX + NPS4**: Documentation states that **localized HBM access** can yield approximately **5-10%** higher **memory bandwidth** (compared to certain default/interleaved configurations); actual results depend on platform BIOS / system settings and real measurements.

When tuning persistent kernels, multi-process, or multi-instance deployments, re-benchmark under the target **partition + NPS** combination.

| Metric | Good | Needs Improvement |
|--------|------|-------------------|
| Occupancy | > 50% | < 25% |
| HBM Bandwidth Utilization | > 60% (memory-bound scenarios) | < 30% |

## Hardware-Specific Tips

- **304 CUs** provide high parallelism; very small grids may underutilize the device — consider persistent kernels or merged launches.
- **64-wide wavefront**: Reduction shuffle step counts differ from 32-wide; do not directly apply NVIDIA warp assumptions.
- **Cache hierarchy**: L3 is 256 MiB (Infinity Cache), L2 total capacity is 32 MiB; tile traversal order significantly impacts L2/L3 reuse — do not confuse with older documentation claiming "L2 = 256 MB".
- **MFMA**: Matrix core and peak values follow official TFLOPS ratings; during tuning, use `rocprof`/Nsight-class tools to distinguish memory-bound vs compute-bound.

## System Tuning (MI300X Host Configuration)

### BIOS Settings (EPYC 9004 Platform)

| Setting | Recommended Value | Reason |
|---------|-------------------|--------|
| Above 4G Decoding | Enabled | Required for GPU BAR |
| SR-IOV | Enabled | Virtualization support |
| SMT Control | Disable | Better for compute-bound workloads |
| NUMA Nodes per Socket | Auto (NPS1) | Default memory interleaving |
| xGMI Max Speed | 32 Gbps | Full fabric bandwidth |
| IOMMU | Enabled | Required for ROCm |
| cTDP / Package Power | 400 W | Maximum sustained power |
| xGMI Force Link Width | x16 | Full link bandwidth |
| APBDIS | 1 | Disable DF P-states for consistency |
| Fixed SOC P-state | P0 | Maximum SoC performance |
| TSME (Memory Encryption) | Disabled | Avoid encryption overhead |

### OS and Runtime Tuning

```bash
# GRUB settings
# Add to kernel command line: pci=realloc=off iommu=pt

# Disable CPU C2 states (low-latency)
cpupower idle-set -d 2

# Disable NUMA auto-balancing
echo 0 > /proc/sys/kernel/numa_balancing

# Device kernel arguments (2-3 us improvement per launch; default since ROCm 6.2)
export HIP_FORCE_DEV_KERNARG=1

# Prevent ROCm thread affinity sprawl
export HSA_OVERRIDE_CPU_AFFINITY_DEBUG=0

# Deterministic GPU clock (reduces PCC events and variance; max 1900 MHz)
rocm-smi --setperfdeterminism 1900
```

### Multi-GPU (RCCL) Tuning

```bash
# MI300X recommended RCCL channels
export NCCL_MIN_NCHANNELS=112

# For CPX partition mode
export TORCH_NCCL_USE_TENSOR_REGISTER_ALLOCATOR_HOOK=1
export RCCL_MSCCLPP_THRESHOLD=$((2*1024*1024*1024))
export MSCCLPP_READ_ALLRED=1
```

## Known Issues

- Omitting `-O3` from `hipcc` may silently fall back to slower instruction modes.
- Triton ROCm backend support for `tl.inline_asm_elementwise` and similar capabilities varies by version — check release notes.
- Grid launch limits differ from NVIDIA — use `hipDeviceGetAttribute` to query upper bounds.
