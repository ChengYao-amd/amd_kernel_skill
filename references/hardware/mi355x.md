# MI355X（GFX950 / CDNA4）优化指南

同规格产品线：**MI350X** 与 MI355X 均为 GFX950 / CDNA4，下列核心参数一致；差异以机型定位与产品文档为准。

## 核心参数

| 参数 | 值 |
|------|-----|
| 架构 | CDNA4 |
| LLVM / offload-arch | `gfx950` |
| 计算单元 (CU) | **256**（8 XCD × 32 CU/XCD） |
| Stream Processors | **16,384** |
| Matrix Cores | **1024**（4 / CU × 256 CU） |
| 晶体管 | **185 B** |
| HBM | 288 GiB HBM3E |
| 内存带宽（峰值） | **8 TB/s** |
| Peak Engine Clock | **2400 MHz** |
| TBP（典型板卡功耗） | **1400 W** |
| Wavefront 大小 | 64 |
| LDS / CU | **160 KiB**（相对 CDNA3 显著增大） |
| L3 Cache | 256 MiB |
| L2 Cache | 32 MiB（每 XCD 4 MiB，共 8 XCD） |
| L1 Vector Cache | 32 KiB / CU |
| VGPR File / CU | 512 KiB |
| SGPR File / CU | 12.5 KiB |
| Infinity Fabric | **7** 条链路，各 **153 GB/s** |
| Scale-out 链路 | **128 GB/s** |
| 形态 | **OAM Module** |
| PCIe | **PCIe 5.0 x16** |
| RAS / 虚拟化 | **RAS**；**Page Retirement**、**Page Avoidance**；**SR-IOV** 支持 |
| Matrix Core 峰值（官方标称，dense） | FP16 **2.5 PF**；FP8 **5 PF**；FP6 **10 PF**；FP4 **10 PF** |
| 数据类型与格式 | 新增 **FP6、FP4**；**block-scaled MFMA（MXFP）** |
| FP8 格式 | **OCP**（E4M3FN + E5M2），**非** CDNA3 上常见的 FNUZ 约定 — 移植时须核对数值与指令语义 |

### 结构化稀疏（Structured Sparsity）与矩阵精度峰值

**Structured sparsity** 可将有效吞吐视为约 **翻倍**（相对 dense 标称）。官方口径示例：

| 模式 | Dense | With structured sparsity |
|------|--------|---------------------------|
| FP16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| FP8 Matrix | 5 PFLOPs | **10.1 PFLOPs** |
| BF16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| INT8 Matrix | 5 POPs | **10.1 POPs** |

算子与库路径若支持 **structured sparsity**，应显式开启并核对数值约束；否则按 dense 峰值做预期管理。

### 变体说明：MI350X

MI350X 与 MI355X 在 **GFX950 / CDNA4** 架构层面规格一致（256 CU、160 KiB LDS/CU、288 GiB HBM3E、同上 Matrix 峰值与数据类型）。具体 SKU、功耗与机框以 AMD 产品页为准。

## 编译

```bash
hipcc -O3 --offload-arch=gfx950 kernel.cpp
```

## 推荐配置

| 路径 | 参数 | 推荐值 |
|------|------|--------|
| Triton | BLOCK_SIZE | 在 160 KiB LDS 前提下可尝试更大 tile，仍以 profiling 为准 |
| Triton | num_warps | 4–8（按寄存器与 LDS 占用迭代） |
| HIP | Thread block | 256 线程起调，结合 LDS 使用加大共享数据结构时重测 |
| MFMA / MXFP | 数据类型 | 显式使用 FP6/FP4/MXFP 路径时核对 ROCm/LLVM 与库版本 |

## 性能阈值参考

| 指标 | 良好 | 需改进 |
|------|------|--------|
| Occupancy | > 50% | < 25% |
| HBM 带宽利用率 | > 60%（内存受限场景） | < 30% |
| MFMA / MXFP 利用率 | > 70%（计算受限场景） | < 40% |
| LDS Bank Conflict | < 5% | > 20% |
| 寄存器溢出 | 0 scratch waves | 任何 scratch |

## 硬件特有技巧

- **160 KiB LDS/CU**：可容纳更大 tile 或更复杂中间结果，但需重新做 occupancy 与 bank conflict 分析，勿沿用 CDNA3 的固定 block 形状。
- **64-bank LDS（CDNA4）**：相对 CDNA3 的 **32 bank** 翻倍，**padding / swizzle 公式需按 64 bank 重算**（见 `isa/memory-instructions.md`）。
- **FP6 / FP4 与 MXFP**：新精度与 block scaling 路径与 CDNA3 MFMA 不同，算子选择与数值范围需对照 ISA 与 ROCm 编程指南。
- **FP8 OCP vs FNUZ**：从 MI300X/MI325X（CDNA3）迁移时，务必区分 FP8 编码与指令，避免静默数值错误。
- **256 CU vs 304 CU（CDNA3）**：grid 划分与负载均衡策略需按 CU 数重算。

## 与 MI300X（GFX942）的差异摘要

| 方面 | MI300X / MI325X (`gfx942`) | MI355X / MI350X (`gfx950`) |
|------|---------------------------|---------------------------|
| CU 数 | 304 | 256 |
| LDS bank | 32 | **64** |
| LDS / CU | 64 KiB | 160 KiB |
| FP8 | CDNA3 生态常见 FNUZ 语境 | **OCP（E4M3FN + E5M2）** |
| 新能力 | — | FP6、FP4、MXFP（block-scaled MFMA） |
| offload-arch | `gfx942` | `gfx950` |

## 从 MI300X 迁移

1. 将 `--offload-arch=gfx942` 改为 `--offload-arch=gfx950`。
2. 按 **256 CU、160 KiB LDS** 重新调优 tile / block。
3. 若使用 FP8，核对 **OCP** 与旧内核中的格式假设。
4. 评估 FP6/FP4/MXFP 是否可替代部分 FP8/BF16 路径。
5. 在团队 `performance_trend.md` 或等价文档中标注硬件代际。

## 已知问题

- `hipcc` 省略 `-O3` 时可能静默回退到较慢的指令模式。
- Triton / 第三方库对 gfx950、MXFP、OCP FP8 的支持随 ROCm 版本变化，升级前后需回归测试。
- 迭代中发现的硬件或驱动问题可记录于此并回溯 `agent_output/` 复盘文档。
