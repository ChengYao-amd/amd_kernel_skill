# MI355X（GFX950 / CDNA4）优化指南

同规格产品线：**MI350X** 与 MI355X 均为 **GFX950 / CDNA4**，**ISA、CU 布局与互连架构**一致；**时钟、TBP、散热与 dense Matrix PF** 因 SKU 而异（见 **Table 2**）。

## 核心参数

| 参数 | 值 |
|------|-----|
| 架构 | CDNA4 |
| LLVM / offload-arch | `gfx950` |
| 计算单元 (CU) | **256**（8 XCD × **32** active CU/XCD；每 XCD 共 **36** CU，**4** disabled 用于良率；**4** 组 × **9** CU，白皮书 **非** CDNA3 的 38/XCD） |
| Stream Processors | **16,384** |
| Matrix Cores | **1024**（4 / CU × 256 CU） |
| 晶体管 | **185 B** |
| HBM | **288 GiB** HBM3E（**8 Gbps**，峰值 **8 TB/s**） |
| IOD | **2**（相对 CDNA3 的 **4** IOD；die-to-die 直连约 **快 14%**） |
| 内存带宽（峰值） | **8 TB/s** |
| Peak Engine Clock | **2400 MHz**（MI355X；MI350X 见下表） |
| TBP（典型板卡功耗） | **1400 W**（MI355X；MI350X 见下表） |
| Wavefront 大小 | 64 |
| LDS / CU | **160 KiB**，**64** banks，**256 B/clock** 读带宽；可从 **L1 data cache** 直载（减轻 VGPR 压力） |
| L3 Cache（Infinity Cache） | **256 MiB**，memory-side；每 stack **16** channels |
| L2 Cache | 32 MiB（每 XCD **4 MiB**，共 8 XCD）；**16-way**，**16** channels/XCD |
| L1 Vector / Data | **32 KiB**/CU；**128 B** cache line；**64-way**（与 CDNA3 一致） |
| L1 Instruction Cache | **64 KiB** / **2** 相邻 CU 共享；**8-way** set-associative |
| VGPR File / CU | 512 KiB |
| SGPR File / CU | 12.5 KiB |
| Infinity Fabric | **8** 条链路，**16-bit** wide，**38.4 Gbps**（较 CDNA3 **32 Gbps** 约 **+20%**）；每链路双向 **76.8 GB/s**（CDNA3 为 **64 GB/s**）；P2P ring 合计 **1075.2 GB/s**，aggregate **1203.2 GB/s** |
| Scale-out 链路 | **128 GB/s** |
| 形态 | **OAM Module** |
| PCIe | **PCIe 5.0 x16** |
| RAS / 虚拟化 | **RAS**；**Page Retirement**、**Page Avoidance**；**SR-IOV** 支持 |
| Matrix Core 峰值（官方标称，dense） | FP16 **2.5 PF**；FP8 **5 PF**；FP6 **10 PF**；FP4 **10 PF**（MI355X；MI350X 见下表） |
| TF32 | **硬件已移除**；由软件以 **BF16** 路径模拟 |
| 超越函数吞吐 | 相对前代约 **2×**（attention 等场景，白皮书） |
| 数据类型与格式 | 新增 **FP6、FP4**；**block-scaled MFMA（MXFP）** |
| FP8 格式 | **OCP**（E4M3FN + E5M2），**非** CDNA3 上常见的 FNUZ 约定 — 移植时须核对数值与指令语义 |

### 每时钟每 CU 算力（FLOPS/clock/CU，CDNA4 / MI355X 列，官方 Table 1）

| Computation | MI300X | MI355X |
|-------------|--------|--------|
| Vector FP64 | 128 | 128 |
| Vector FP32 | 256 | 256 |
| Vector FP16 | 256 | 256 |
| Matrix FP64 | 256 | **128**（减半） |
| Matrix FP32 | 256 | 256 |
| Matrix FP16 / Sparsity | 2048 | **4096**（翻倍） |
| Matrix BF16 / Sparsity | 2048 | **4096** |
| Matrix FP8 / Sparsity | 4096 | **8192**（翻倍） |
| Matrix INT8 / Sparsity | 4096 | **8192** |
| Matrix MXFP6 | N/A | **16384**（新） |
| Matrix MXFP4 | N/A | **16384**（新） |

说明：**Matrix FP64** 在 MI355X 上 per-clock/CU 峰值相对 MI300X **降低**；**FP16/BF16/FP8/INT8 Matrix（含 sparsity）** 及 **MXFP** 路径峰值 **升高**，与产品级 PF 标称及 **structured sparsity 翻倍** 叙述一致，调优时勿混用「向量」与「矩阵」口径。

### MI350X vs MI355X（官方 Table 2 摘要）

| | MI350X | MI355X |
|---|--------|--------|
| Clock | **2200 MHz** | **2400 MHz** |
| TBP | **1000 W** | **1400 W** |
| Cooling | Air（被动） | Liquid（DLC） |
| FP16 Matrix | **2.3 PF** | **2.5 PF** |
| FP8 Matrix | **4.6 PF** | **5.0 PF** |
| MXFP6 / MXFP4 | **9.2 PF** | **10 PF** |

### 结构化稀疏（Structured Sparsity）与矩阵精度峰值

**Structured sparsity** 对 **FP16 / BF16 / FP8 / INT8** 可将有效吞吐视为约 **翻倍**（相对 dense 标称；与上表 Matrix sparsity 列一致）。官方口径示例：

| 模式 | Dense | With structured sparsity |
|------|--------|---------------------------|
| FP16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| FP8 Matrix | 5 PFLOPs | **10.1 PFLOPs** |
| BF16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| INT8 Matrix | 5 POPs | **10.1 POPs** |

算子与库路径若支持 **structured sparsity**，应显式开启并核对数值约束；否则按 dense 峰值做预期管理。

### 变体说明：MI350X

MI350X 与 MI355X 在 **GFX950 / CDNA4** 架构层面一致（256 active CU、XCD 36/32 布局、160 KiB LDS/CU、288 GiB HBM3E、数据类型与 ISA）。**时钟、TBP、散热与 dense Matrix PF** 以 **Table 2** 为准（见上表）；机框与认证以 AMD 产品页为准。

## Partition 与 NPS（CDNA4 更新）

| 模式 | 配置 | 每 partition 显存（约） |
|------|------|-------------------------|
| **SPX** | **8** XCD 全卡 | **288 GB** |
| **DPX** | **4** XCD / partition，**2** partitions | **144 GB** each |
| **QPX**（CDNA4 新增） | **2** XCD / partition，**4** partitions | **72 GB** each |
| **CPX** | **1** XCD / partition，**8** partitions | **36 GB** each |

**NPS**：仅 **NPS1**（交错）或 **NPS2**（**per-IOD**）；白皮书称 **DPX + NPS2** 在多数场景 **能效/局部性最优**（相对其他组合以实测为准）。

## 编译

```bash
hipcc -O3 --offload-arch=gfx950 kernel.cpp
```

## 推荐配置

| 路径 | 参数 | 推荐值 |
|------|------|--------|
| Triton | BLOCK_SIZE | 在 160 KiB LDS 前提下可尝试更大 tile，仍以 profiling 为准 |
| Triton | num_warps | 4–8（按寄存器与 LDS 占用迭代） |
| HIP | Thread block | 256 线程起调，结合 LDS 使用加大共享数据结构时重测 |
| MFMA / MXFP | 数据类型 | 显式使用 FP6/FP4/MXFP 路径时核对 ROCm/LLVM 与库版本 |

## 性能阈值参考

| 指标 | 良好 | 需改进 |
|------|------|--------|
| Occupancy | > 50% | < 25% |
| HBM 带宽利用率 | > 60%（内存受限场景） | < 30% |
| MFMA / MXFP 利用率 | > 70%（计算受限场景） | < 40% |
| LDS Bank Conflict | < 5% | > 20% |
| 寄存器溢出 | 0 scratch waves | 任何 scratch |

## 硬件特有技巧

- **160 KiB LDS/CU**：可容纳更大 tile 或更复杂中间结果，但需重新做 occupancy 与 bank conflict 分析，勿沿用 CDNA3 的固定 block 形状。
- **64-bank LDS（CDNA4）**：相对 CDNA3 的 **32 bank** 翻倍，**padding / swizzle 公式需按 64 bank 重算**（见 `isa/memory-instructions.md`）。
- **FP6 / FP4 与 MXFP**：新精度与 block scaling 路径与 CDNA3 MFMA 不同，算子选择与数值范围需对照 ISA 与 ROCm 编程指南。
- **FP8 OCP vs FNUZ**：从 MI300X/MI325X（CDNA3）迁移时，务必区分 FP8 编码与指令，避免静默数值错误。
- **256 CU vs 304 CU（CDNA3）**：grid 划分与负载均衡策略需按 CU 数重算。

## 与 MI300X（GFX942）的差异摘要

| 方面 | MI300X / MI325X (`gfx942`) | MI355X / MI350X (`gfx950`) |
|------|---------------------------|---------------------------|
| XCD / CU | 每 XCD **40** CU（**38** active）；**8** XCD → **304** CU | 每 XCD **36** CU（**32** active）；**8** XCD → **256** CU |
| IOD | **4** | **2**（直连延迟/带宽特性不同） |
| Infinity Fabric | **32 Gbps**/link 等（见 MI300X 文档） | **38.4 Gbps**/link；每链路双向 **76.8 GB/s** |
| CU 数（active） | 304 | 256 |
| LDS bank | 32 | **64** |
| LDS / CU | 64 KiB | 160 KiB |
| Matrix FP64（FLOPS/clock/CU） | 256 | **128** |
| TF32 | 以 CDNA3 ISA/栈为准（本文未展开） | **硬件无**；**BF16** 软件路径 |
| FP8 | CDNA3 生态常见 FNUZ 语境 | **OCP（E4M3FN + E5M2）** |
| 新能力 | — | FP6、FP4、MXFP（block-scaled MFMA）；**QPX** partition |
| offload-arch | `gfx942` | `gfx950` |

## 从 MI300X 迁移

1. 将 `--offload-arch=gfx942` 改为 `--offload-arch=gfx950`。
2. 按 **256 CU、160 KiB LDS** 重新调优 tile / block。
3. 若使用 FP8，核对 **OCP** 与旧内核中的格式假设。
4. 评估 FP6/FP4/MXFP 是否可替代部分 FP8/BF16 路径。
5. 在团队 `performance_trend.md` 或等价文档中标注硬件代际。

## 已知问题

- `hipcc` 省略 `-O3` 时可能静默回退到较慢的指令模式。
- Triton / 第三方库对 gfx950、MXFP、OCP FP8 的支持随 ROCm 版本变化，升级前后需回归测试。
- 迭代中发现的硬件或驱动问题可记录于此并回溯 `agent_output/` 复盘文档。
