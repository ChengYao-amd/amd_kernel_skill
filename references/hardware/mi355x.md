# MI355X (GFX950 / CDNA4) Optimization Guide

Same-spec product line: **MI350X** and MI355X are both **GFX950 / CDNA4**, with identical **ISA, CU layout, and interconnect architecture**; **clock speed, TBP, cooling, and dense Matrix PF** vary by SKU (see **Table 2**).

## Core Parameters

| Parameter | Value |
|-----------|-------|
| Architecture | CDNA4 |
| LLVM / offload-arch | `gfx950` |
| Compute Units (CU) | **256** (8 XCD x **32** active CU/XCD; each XCD has **36** CU total, **4** disabled for yield; **4** groups x **9** CU, per white paper — **not** the same as CDNA3's 38/XCD) |
| Stream Processors | **16,384** |
| Matrix Cores | **1024** (4 / CU x 256 CU) |
| Transistors | **185 B** |
| HBM | **288 GiB** HBM3E (**8 Gbps**, peak **8 TB/s**) |
| IOD | **2** (compared to CDNA3's **4** IOD; die-to-die direct connection approximately **14% faster**) |
| Memory Bandwidth (peak) | **8 TB/s** |
| Peak Engine Clock | **2400 MHz** (MI355X; see table below for MI350X) |
| TBP (typical board power) | **1400 W** (MI355X; see table below for MI350X) |
| Wavefront Size | 64 |
| LDS / CU | **160 KiB**, **64** banks, **256 B/clock** read bandwidth; can load directly from **L1 data cache** (reduces VGPR pressure) |
| L3 Cache (Infinity Cache) | **256 MiB**, memory-side; **16** channels per stack |
| L2 Cache | 32 MiB (**4 MiB** per XCD, 8 XCDs total); **16-way**, **16** channels/XCD |
| L1 Vector / Data | **32 KiB**/CU; **128 B** cache line; **64-way** (same as CDNA3) |
| L1 Instruction Cache | **64 KiB** / shared by **2** adjacent CUs; **8-way** set-associative |
| VGPR File / CU | 512 KiB |
| SGPR File / CU | 12.5 KiB |
| Infinity Fabric | **8** links, **16-bit** wide, **38.4 Gbps** (approximately **+20%** over CDNA3's **32 Gbps**); per-link bidirectional **76.8 GB/s** (CDNA3: **64 GB/s**); P2P ring total **1075.2 GB/s**, aggregate **1203.2 GB/s** |
| Scale-out Links | **128 GB/s** |
| Form Factor | **OAM Module** |
| PCIe | **PCIe 5.0 x16** |
| RAS / Virtualization | **RAS**; **Page Retirement**, **Page Avoidance**; **SR-IOV** support |
| Matrix Core Peak (official rated, dense) | FP16 **2.5 PF**; FP8 **5 PF**; FP6 **10 PF**; FP4 **10 PF** (MI355X; see table below for MI350X) |
| TF32 | **Removed from hardware**; emulated in software via **BF16** path |
| Transcendental Function Throughput | Approximately **2x** over previous generation (attention-class scenarios, per white paper) |
| Data Types and Formats | New **FP6, FP4**; **block-scaled MFMA (MXFP)** |
| FP8 Format | **OCP** (E4M3FN + E5M2), **not** the FNUZ convention common on CDNA3 — verify numerical values and instruction semantics when porting |

### Per-Clock Per-CU Throughput (FLOPS/clock/CU, CDNA4 / MI355X column, official Table 1)

| Computation | MI300X | MI355X |
|-------------|--------|--------|
| Vector FP64 | 128 | 128 |
| Vector FP32 | 256 | 256 |
| Vector FP16 | 256 | 256 |
| Matrix FP64 | 256 | **128** (halved) |
| Matrix FP32 | 256 | 256 |
| Matrix FP16 / Sparsity | 2048 | **4096** (doubled) |
| Matrix BF16 / Sparsity | 2048 | **4096** |
| Matrix FP8 / Sparsity | 4096 | **8192** (doubled) |
| Matrix INT8 / Sparsity | 4096 | **8192** |
| Matrix MXFP6 | N/A | **16384** (new) |
| Matrix MXFP4 | N/A | **16384** (new) |

Note: **Matrix FP64** per-clock/CU peak on MI355X is **reduced** relative to MI300X; **FP16/BF16/FP8/INT8 Matrix (with sparsity)** and **MXFP** path peaks are **increased**, consistent with product-level PF ratings and the **structured sparsity doubling** narrative. Do not conflate "vector" and "matrix" metrics when tuning.

### MI350X vs MI355X (Official Table 2 Summary)

| | MI350X | MI355X |
|---|--------|--------|
| Clock | **2200 MHz** | **2400 MHz** |
| TBP | **1000 W** | **1400 W** |
| Cooling | Air (passive) | Liquid (DLC) |

| FP8 Matrix | **4.6 PF** | **5.0 PF** |

### Structured Sparsity and Matrix Precision Peaks

**Structured sparsity** can approximately **double** effective throughput for **FP16 / BF16 / FP8 / INT8** (relative to dense rated values; consistent with the Matrix sparsity columns in the table above). Official examples:

| Mode | Dense | With structured sparsity |
|------|--------|---------------------------|
| FP16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| FP8 Matrix | 5 PFLOPs | **10.1 PFLOPs** |
| BF16 Matrix | 2.5 PFLOPs | **5 PFLOPs** |
| INT8 Matrix | 5 POPs | **10.1 POPs** |

If operator and library paths support **structured sparsity**, explicitly enable it and verify numerical constraints; otherwise manage expectations based on dense peak values.

### Variant Notes: MI350X

MI350X and MI355X are architecturally identical at the **GFX950 / CDNA4** level (256 active CU, XCD 36/32 layout, 160 KiB LDS/CU, 288 GiB HBM3E, data types and ISA). **Clock speed, TBP, cooling, and dense Matrix PF** follow **Table 2** (see table above); chassis and certifications per AMD product page.

## Partition and NPS (CDNA4 Updates)

| Mode | Configuration | VRAM per Partition (approx.) |
|------|---------------|------------------------------|
| **SPX** | **8** XCD full card | **288 GB** |
| **DPX** | **4** XCD / partition, **2** partitions | **144 GB** each |
| **QPX** (new in CDNA4) | **2** XCD / partition, **4** partitions | **72 GB** each |
| **CPX** | **1** XCD / partition, **8** partitions | **36 GB** each |

**NPS**: Only **NPS1** (interleaved) or **NPS2** (**per-IOD**); the white paper states **DPX + NPS2** is **optimal for energy efficiency/locality** in most scenarios (verify with real measurements for other combinations).

## Performance Threshold Reference

| Metric | Good | Needs Improvement |
|--------|------|-------------------|
| Occupancy | > 50% | < 25% |
| HBM Bandwidth Utilization | > 60% (memory-bound scenarios) | < 30% |
| MFMA / MXFP Utilization | > 70% (compute-bound scenarios) | < 40% |
| LDS Bank Conflict | < 5% | > 20% |
| Register Spill | 0 scratch waves | Any scratch |

## Hardware-Specific Tips

- **160 KiB LDS/CU**: Can accommodate larger tiles or more complex intermediate results, but requires fresh occupancy and bank conflict analysis — do not reuse CDNA3's fixed block shapes.
- **64-bank LDS (CDNA4)**: Doubled from CDNA3's **32 banks**; **padding / swizzle formulas must be recalculated for 64 banks** (see `isa/memory-instructions.md`).
- **FP6 / FP4 and MXFP**: New precision and block scaling paths differ from CDNA3 MFMA; operator selection and numerical ranges must reference ISA and ROCm programming guides.
- **FP8 OCP vs FNUZ**: When migrating from MI300X/MI325X (CDNA3), be sure to distinguish FP8 encoding and instructions to avoid silent numerical errors.
- **256 CU vs 304 CU (CDNA3)**: Grid partitioning and load balancing strategies must be recalculated for the CU count.

## Differences from MI300X (GFX942) Summary

| Aspect | MI300X / MI325X (`gfx942`) | MI355X / MI350X (`gfx950`) |
|--------|---------------------------|---------------------------|
| XCD / CU | **40** CU per XCD (**38** active); **8** XCD -> **304** CU | **36** CU per XCD (**32** active); **8** XCD -> **256** CU |
| IOD | **4** | **2** (different direct-connection latency/bandwidth characteristics) |
| Infinity Fabric | **32 Gbps**/link, etc. (see MI300X docs) | **38.4 Gbps**/link; per-link bidirectional **76.8 GB/s** |
| Active CU Count | 304 | 256 |
| LDS banks | 32 | **64** |
| LDS / CU | 64 KiB | 160 KiB |
| Matrix FP64 (FLOPS/clock/CU) | 256 | **128** |
| TF32 | Per CDNA3 ISA/stack (not expanded here) | **No hardware support**; **BF16** software path |
| FP8 | CDNA3 ecosystem commonly uses FNUZ context | **OCP (E4M3FN + E5M2)** |
| New Capabilities | — | FP6, FP4, MXFP (block-scaled MFMA); **QPX** partition |
| offload-arch | `gfx942` | `gfx950` |

## Migrating from MI300X

1. Change `--offload-arch=gfx942` to `--offload-arch=gfx950`.
2. Re-tune tile / block sizes for **256 CU, 160 KiB LDS**.
3. If using FP8, verify **OCP** against format assumptions in existing kernels.
4. Evaluate whether FP6/FP4/MXFP can replace some FP8/BF16 paths.
5. Note the hardware generation in your team's `performance_trend.md` or equivalent documentation.

## Known Issues

- Omitting `-O3` from `hipcc` may silently fall back to slower instruction modes.
- Triton / third-party library support for gfx950, MXFP, and OCP FP8 varies by ROCm version — run regression tests before and after upgrades.
- Hardware or driver issues discovered during iteration can be logged here and cross-referenced with `agent_output/` postmortem documents.
