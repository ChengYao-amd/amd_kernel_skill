# GEMM Tuning Comprehensive Guide

This document provides a **layered GEMM tuning path**, library-level **APIs**, **Stream-K**, **Online Tuning**, **Tensile** parameters, **Attention** implementation selection, and published performance data for **AMD Instinct** GPUs.

---

## Three-Level GEMM Tuning Strategy (Top-Down)

| Level | Approach | Description |
|-------|----------|-------------|
| **1. Pre-tuned Docker / Preset Configuration Tables** | Official inference images (e.g., `rocm/sgl-dev:*`), AITER `gradlib`-produced `*_tuned_gemm.csv`, repository-bundled **CSV/JSON** per model | Reproduce the environment and known optimal **shape -> kernel** mappings, avoiding cold-start full-range search. |
| **2. PyTorch TunableOp -> hipBLASLt / rocBLAS** | `PYTORCH_TUNABLEOP_ENABLED=1` drives runtime selection of the best candidate from **rocBLAS** / **hipBLASLt** and persists the **GEMM Table** to disk | Automatically adapts within the framework to current hardware and precision; suitable for long-term reuse in training/inference. |
| **3. TensileLite / Tensile Custom Kernels** | **Tensile** generates kernels described via **YAML** for **GEMM** variants, loaded by **rocBLAS** / **hipBLASLt** | Used when covering **shapes** not optimized by the library, special **epilogues**, or new **dtypes**; high iteration cost but also the highest ceiling. |

**Relationship summary**: Low-level **GEMM** is mostly generated by **Tensile**; **rocBLAS** exposes `rocblas_*gemm*`; **hipBLASLt** provides richer fusion and **algorithm** selection; **PyTorch** uses **TunableOp** to choose the best among them (and table-driven results).

## hipBLASLt API Workflow

Core operation: \(D = \text{Activation}(\alpha \cdot op(A) \cdot op(B) + \beta \cdot op(C) + \text{bias})\).

Recommended calling sequence:

1. `hipblasLtCreate()` -- Create **library handle**
2. `hipblasLtMatrixLayoutCreate()` -- **A/B/C/D** **layout** (type, rows and columns, **ld**)
3. `hipblasLtMatmulDescCreate()` -- **matmul descriptor** (`computeType`, `scaleType`, etc.)
4. `hipblasLtMatmulDescSetAttribute()` -- **transpose**, **epilogue**, **bias**, etc. (`hipblasLtMatmulDescAttributes_t`)
5. `hipblasLtMatmulPreferenceCreate()` -- **workspace** limit and other preferences
6. `hipblasLtMatmulAlgoGetHeuristic()` -- **algorithm** list sorted by estimated latency (`requestedAlgoCount`)
7. `hipblasLtMatmul()` -- Execute **GEMM** (`alpha`, `beta`, `algo`, `workspace`, `stream`)
8. `hipblasLtDestroy()` -- Release resources

Commonly used function quick reference: `hipblasLtMatmulDescCreate`, `hipblasLtMatmulAlgoGetHeuristic`, `hipblasLtMatmul`.

## rocBLAS GEMM Entry Points & Tuning Relationship

**rocBLAS** is the foundational **BLAS** for **ROCm**; **Level-3 GEMM** internally delegates to **Tensile**. Naming convention: `rocblas_<type>gemm()`.

| Entry Point Example | Use Case |
|---------------------|----------|
| `rocblas_sgemm` | **FP32** |
| `rocblas_dgemm` | **FP64** |
| `rocblas_hgemm` | **FP16** |
| `rocblas_gemm_ex` | Mixed precision (**FP8**, **BF16**, **INT8**, etc.) |
| `rocblas_*gemm_batched` / `*strided_batched` | **Batched** scenarios |

On the tuning side: besides directly switching **APIs**, this is typically combined with **`TENSILE_*` environment variables**, **TunableOp**, or **Tensile** logic files (see below) to change the actually selected kernel.

## Stream-K Scheduling (`TENSILE_SOLUTION_SELECTION_METHOD=2`)

**Stream-K** partitions the inner **K**-dimension iterations across **CUs**, benefiting non-square and load-imbalanced shapes; on the **MI350** series it is the **only** strategy.

| Environment Variable | Values & Meaning |
|---------------------|-----------------|
| `TENSILE_SOLUTION_SELECTION_METHOD` | `0` = standard **tuned** (default); `2` = **Stream-K** |
| `TENSILE_STREAMK_DYNAMIC_GRID` | `6` = auto (default); `0` = use all **CUs** |
| `TENSILE_STREAMK_FIXED_GRID` | Integer: fixed **workgroup** count |
| `TENSILE_STREAMK_MAX_CUS` | Upper-limit **CU** count, used to cap **Stream-K** occupancy |

**Concurrency hint**: For example, setting `TENSILE_STREAMK_FIXED_GRID=64` can prevent a single **GEMM** from fully occupying the **GPU**, facilitating co-residency with other kernels.

## hipBLASLt Online Tuning (`HIP_ONLINE_TUNING=1`)

Enables **hipBLASLt** **GEMM online tuning** (commonly used on the **gradlib** / toolchain side):

```bash
export HIP_ONLINE_TUNING=1
```

The first run incurs a one-time overhead of several minutes; results can be written to **`hip_online_tuning_res.csv`** for subsequent loading. Suitable for scenarios where **shape** distributions vary widely and you want to converge online to the optimal **solution** for the current machine.

## Key Tensile / TensileLite Parameters (Concepts)

In **Tensile** logic and **YAML**, the following items collectively determine **macro** and **instruction** selection (names may vary slightly across versions; refer to the **Tensile** branch in use):

| Parameter Class | Function |
|----------------|----------|
| **WorkGroup** | **Workgroup** grid and the **tile** range each **block** is responsible for |
| **ThreadTile** | **Thread**-level **M x N** sub-block, affecting register and **lane** mapping |
| **MacroTile** | **Block**-level large **tile** (commonly corresponds to the main partitioning in the **M/N/K** directions) |
| **LoopUnroll** | **K**-loop unroll depth, balancing **ILP** and register pressure |
| **SplitU / GlobalSplitU** | **K**-dimension splitting and multi-**wave** reduction strategy |
| **MatrixInstruction** | The **MFMA** (or other **Matrix Core**) instruction shape and type to use |

**YAML** example (from crawl reports): `SGEMM{M: 5504, N: 5504, K: 5504, transposeA: false, transposeB: true, dataType: S}`; mixed precision can be written as **`GEMM_EX (HHS){...}`**, etc.

## Attention Kernel Selection Guide

Combining latency, kernel maturity, and **ROCm** support, the typical priority (requires empirical testing for the same scenario):

**AITER asm (assembly and other extreme paths) > CK FMHA (Composable Kernel / CK-Tile) > Triton FA > FlashInfer**

| Basis | Description |
|-------|-------------|
| **AITER** | **inline asm**, **CK**, **Triton** multi-path; blog posts on **DeepSeek** and similar scenarios show significant **speedup** relative to baseline (see table below). |
| **CK-Tile** | **FA-v2** can be expressed in roughly a hundred lines, with clear **MFMA** and **LDS** pipeline. |
| **Triton** | Fast development iteration; on **MI300X**, **TileLang** blog posts show it is often weaker than optimal **CK/handwritten** under the same configuration. |
| **FlashInfer** | Service-oriented **paged KV**, **GQA/MQA**, **torch.compile**; multi-version **ROCm** support, usable as a general-purpose backend. |

## TensileLite: Custom Kernel Generation for Specific GEMM Shapes

When the pre-existing kernel pool does not cover a target shape (common for skinny-M decode shapes in LLM inference), **TensileLite** generates entirely new Tensile kernels optimized for the exact (M, N, K).

### Three Tuning Approaches Compared

| Approach | Mechanism | Speed | Performance Ceiling |
|----------|-----------|-------|---------------------|
| **Offline Tuning** | Select best from existing kernel pool | Fast | Limited by pool |
| **Online Tuning** | Runtime benchmarking + caching | Medium (warm-up) | Limited by pool |
| **TensileLite** | Generate new kernels for specific shapes | Slow | Near-optimal |

### TensileLite Tuning Parameter Categories

**Thread and Workgroup Organization:**
- `WorkGroup [dim0, dim1, LocalSplitU]` -- workgroup size for occupancy
- `ThreadTile [dim0, dim1]` -- per-thread tile size
- `MacroTile = WorkGroup x ThreadTile`
- `WorkGroupMapping` -- work-group ordering for L2 cache locality

**Loop and Unrolling:**
- `LoopUnroll` -- inner loop unroll factor (higher = more registers, better coalescing)
- `DepthU = LoopUnroll x SplitU`

**Split-K Parallelism:**
- `LocalSplitU` -- split summation within workgroup
- `GlobalSplitU` -- split across workgroups (requires atomic reduction kernel)

**Memory Access:**
- `PrefetchGlobalRead` -- prefetch global data one iteration ahead
- `PrefetchLocalRead` -- prefetch LDS data one iteration ahead
- `VectorWidth` -- vector load size (e.g., float4)

**Instruction:**
- `MatrixInstruction` -- MFMA instruction shape + wave tiling parameters

### TensileLite Tuning Workflow

```bash
# Step 1: Capture GEMM shapes from the application
export HIPBLASLT_LOG_MASK=32
export HIPBLASLT_LOG_FILE=./hipblaslt.log
hipblaslt-bench -m 1280 -n 32 -k 5120 --a_type f16_r ...

# Step 2: Generate Tensile config and run tuning
python tensile_config_generator.py --hipblaslt_log ./hipblaslt.log \
  --tensile_config ./tuning_template.yaml --gpus 4 --iters 100
./TensileLite/build_tmp/Tensile.sh tuning_template.yaml tuning_result

# Step 3: Merge tuned kernels into the library and rebuild
python3 merge.py <existing_logic_dir> <tuning_result/3_LibraryLogic/> <output_dir>
./install.sh -idc -a gfx942
```

### TensileLite Performance Results (MI300X, f16 -> f32)

| Metric | Value |
|--------|-------|
| Average speedup vs baseline | **2.25x** |
| Average speedup vs offline tuning | **1.19x** |
| Best case (skinny M) | **3.2x** vs baseline |
| Example: shape (165, 14400, 120) | 42.3 us vs 135.9 us baseline |

Skinny matrices (small M, typical of LLM decode) benefit most because the default kernel pool is optimized for larger, more square shapes.

## hipBLASLt Online Tuning Workflow Details

### Enable

```bash
export HIP_ONLINE_TUNING=1
# For vLLM integration:
export VLLM_ROCM_USE_AITER_HIP_ONLINE_TUNING=1
```

### How It Works

1. On first encounter of a new GEMM shape, check cache file for a matching config.
2. If cache miss: benchmark all candidate kernels, select the best.
3. Save solution index to cache file for reuse.

### Performance Characteristics

| Metric | Value |
|--------|-------|
| Performance vs baseline | **105.98%** (average) |
| Performance vs offline tuning | **100.79%** (average) |
| One-time overhead per unseen shape | ~31 seconds |

Online tuning is integrated into the AITER framework and is the easiest path to improved GEMM performance without manual intervention.

## Library-Level GEMM Tuning Commands

### rocBLAS (Tensile-based) Tuning

```bash
# Step 1: Capture GEMM calls
ROCBLAS_LAYER=4 ROCBLAS_LOG_PATH=./gemms.yaml ./app

# Step 2: Run tuning
/opt/rocm/bin/rocblas-gemm-tune --yaml gemms.yaml

# Step 3: Apply tuned results
export ROCBLAS_TENSILE_GEMM_OVERRIDE_PATH=result.csv
```

### hipBLASLt (hipblaslt-bench) Tuning

```bash
# Step 1: Capture GEMM calls
HIPBLASLT_LOG_MASK=32 HIPBLASLT_LOG_FILE=log.log ./app

# Step 2: Benchmark candidates
/opt/rocm/bin/hipblaslt-bench --api_method c -m M -n N -k K ...

# Step 3: Apply tuned results
export HIPBLASLT_TUNING_OVERRIDE_FILE=tuning.txt
```

### PyTorch TunableOp

```bash
# Step 1: Tuning pass (slow, run once)
PYTORCH_TUNABLEOP_ENABLED=1 PYTORCH_TUNABLEOP_TUNING=1 your_script.sh

# Step 2: Use tuned results (fast, run always)
PYTORCH_TUNABLEOP_ENABLED=1 PYTORCH_TUNABLEOP_TUNING=0 your_script.sh
```

Typical result: **>20% GEMM improvement** with TunableOp.

## FP8 GEMM Optimization on CDNA4 (MI355X)

### Progressive Optimization Results (M=N=K=4096, FP8 -> BF16, FP32 accumulator)

| Stage | TFLOPS/s | Speedup vs Naive |
|-------|----------|------------------|
| Naive (1 thread/element) | 1.15 | 1x |
| LDS tiling | 4.80 | 4.2x |
| MFMA matrix-core | 30.05 | 26x |
| + Vectorized loads (fp8x16) | 336.88 | 293x |
| + Direct global-to-LDS load | 506.70 | 441x |
| + LDS swizzle + double buffer | 1,166.41 | 1,014x |
| + 256x256 tile, 8 waves | **2,288.16** | **1,990x** |
| hipBLASLt reference (4096) | ~2,750 | -- |
| hipBLASLt reference (8192) | ~3,130 | -- |

### Key CDNA4 Techniques

1. **MFMA 16x16x128 FP8**: 65,536 FLOPs per instruction (512x larger than scalar FMA)
2. **Vectorized FP8 loads**: `fp8x16_t` = 16 FP8 values per load (128 bits)
3. **Direct global-to-LDS**: `llvm_amdgcn_raw_buffer_load_lds()` bypasses registers; 4x wider per lane on CDNA4 (128-bit vs 32-bit)
4. **LDS swizzling**: XOR-based row remap eliminates bank conflicts with 64-bank LDS
5. **Double buffering**: Ping-pong LDS slots overlap load(t+1) with compute(t)
6. **8-wave ping-pong scheduling**: 2 waves/SIMD alternate memory and MMA ops using `s_barrier`, `s_setprio`, `sched_barrier` intrinsics

### Optimal Configuration

256x256 output tile with 512 threads (8 waves) achieves the highest measured performance. 1024-thread variants show diminishing returns from synchronization overhead.
